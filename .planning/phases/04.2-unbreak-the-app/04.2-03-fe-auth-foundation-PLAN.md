<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage → apiFetch | JWT read from storage and injected into request headers |
| apiFetch 401 handler → window.location | Redirect target built from window.location.pathname + search; only the host/path segments are re-used, no external URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-03-01 | Information Disclosure | JWT in localStorage exposed to XSS | accept | D-01 explicitly accepts localStorage for v1. Phase 8 ships CSP + rehype-sanitize. No new XSS surface is introduced by auth.ts itself — the login form renders only fixed copy, no user HTML. |
| T-42-03-02 | Spoofing | Open-redirect via the ?from= parameter in the 401 redirect | mitigate | The from= value is built from window.location.pathname (same-origin) and passed to react-router's useNavigate (Plans 05+) which does not accept absolute URLs. The apiFetch wrapper uses window.location.assign only for the /login redirect itself, not for the from= target. Verify in Plan 05: a from=https://evil.example.com redirect must resolve to /login, not the external URL. |
| T-42-03-03 | Information Disclosure | apiFetch wrapper logs token value in error path | mitigate | apiFetch must not console.log the token or the Authorization header value. Error surfaces via the thrown Error('Session expired') string only. |
| T-42-03-04 | Information Disclosure | providerErrors.ts inlineCode array renders shell commands | accept | Shell commands ('ollama serve', 'ollama pull qwen3:4b', 'OPENAI_API_KEY') are help text rendered in a Chakra <code> element as plain text (no dangerouslySetInnerHTML). WCAG and XSS properties are safe. |
</threat_model>
