# Phase 4.2: Unbreak the App - Research

**Researched:** 2026-05-15
**Domain:** React Router login + JWT-aware fetch + provider-health probe (FastAPI + Chakra UI v3)
**Confidence:** HIGH

## Summary

Phase 4.2 closes two gaps that block in-browser usage: the missing `/login` UI for the
already-shipped JWT backend, and a model selector that creates a session against a provider
whose liveness has never been probed. The shape of both fixes is well-bounded by CONTEXT.md
and UI-SPEC.md — the research below is mostly **verification + integration recipe** for the
locked decisions, plus a tight set of recommendations in the discretion areas (file layout,
auth context, fetch wrapper).

Two findings that diverge from the locked decisions and the planner should resolve before
task generation:

1. **D-17 vs. specifics block:** D-17 calls for a new `GET /api/auth/me`; the `<specifics>`
   block notes `/users/me` already exists at `auth.py:195` and suggests reusing it. Reusing
   it is cheaper and matches the existing surface — the planner should explicitly pick one.
2. **D-14 package name:** "`react-router-dom` v7" is the canonical install, but v7 also
   re-exports the same API from `react-router` (a thin alias). `react-router-dom@7.15.1`
   depends on `react-router@7.15.1`. Either install line works. Recommend keeping
   `react-router-dom` per the locked decision text — search results and existing community
   examples are still indexed against that name.

**Primary recommendation:** Use `react-router-dom@7` declarative mode (`BrowserRouter` +
`Routes` + `Route`); centralize auth in a single `lib/auth.ts` module exposing `getToken()`,
`setToken()`, `clearToken()`, and an `apiFetch()` wrapper that injects `Authorization:
Bearer` on every request and forces a 401-redirect via a window-scoped helper; expose probe
results from `POST /api/chat/session` via a structured `{error, message, hint}` body using
the existing `HTTPException(detail=...)` pattern; reuse the shipped `/users/me` for the
boot-time token-validity check.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Auth Storage & Lifecycle:**

- **D-01: JWT lives in `localStorage` under key `auth_token`.** Mirrors the existing
  `llm_provider_config` localStorage pattern. XSS exposure acknowledged and accepted for v1.
- **D-02: 401 from any `/api/*` call → clear token, redirect to `/login`, flash "Session
  expired — please sign in again."** Centralized in the fetch wrapper.
- **D-03: Storage layout — separate keys: `auth_token` + `llm_provider_config`.** Logout
  clears `auth_token` only.
- **D-04: Token sent as `Authorization: Bearer <jwt>` header on every `/api/*` request.**
- **D-05: Logout clears `auth_token` and any in-flight chat session, then routes to
  `/login`.**

**Selector Failure-Mode UX:**

- **D-06: Errors render under the provider/model selector in the chat header
  (dropdown-adjacent inline).** Persistent, no auto-dismiss.
- **D-07: Errors fire proactively at session-create time** (Ollama: hit `GET
  {ollama_base_url}/api/tags` and verify selected model is in the list; Cloud: env-var
  presence only — no live key validation in v1).
- **D-08: Recovery UX is "suggestion text + Retry button + selector remains live."** No
  auto-fallback.
- **D-09: F1–F4 failure-mode taxonomy locked** (Ollama unreachable, model not pulled, cloud
  API key missing, `/api/providers` failure).

**Login Visual Direction:**

- **D-10: Editorial split-screen for `/login`.** UI-SPEC.md owns visual contract.
- **D-11: Form uses Chakra v3 `Field.Root` + `Field.Label` + `Field.ErrorText`.**
- **D-12: Password field uses an unmasked-toggle (eye icon).**
- **D-13: Failed-login error renders inline under the form** ("Incorrect username or
  password").

**Routing & Protected Routes:**

- **D-14: Use `react-router-dom` v7** (`BrowserRouter` + `Routes` + `Route`).
- **D-15: `<RequireAuth>` wrapper component** checks `auth_token` and `getCurrentUser()`;
  redirects to `/login` with `?from=` param.
- **D-16: Routes: `/login` → `<Login />`, `/` → `<RequireAuth><ChatInterface /></RequireAuth>`.**

**Backend additions:**

- **D-17: New `GET /api/auth/me` endpoint** (uses existing `get_current_active_user`).
  *(Note: `<specifics>` block says `/users/me` already exists — reuse-or-rename decision
  flagged below.)*
- **D-18: Provider health probe exposed via `POST /api/chat/session`.** Structured error
  body `{error: "ollama_unreachable" | "model_not_installed" | "missing_api_key", message,
  hint}`.
- **D-19: `AUTH_USERS` env-seed stays as the user source.**

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Credential collection (username/password form) | Browser / Client | — | Form is purely UI; no secrets at rest beyond the form's transient state. |
| JWT issuance | API / Backend | — | `POST /token` already issues HS256 JWT signed with `JWT_SECRET`. |
| JWT storage | Browser / Client | — | localStorage by D-01. Backend is stateless w.r.t. tokens. |
| JWT injection on `/api/*` | Browser / Client | — | Single fetch wrapper reads the token and adds `Authorization: Bearer …`. |
| JWT validation | API / Backend | — | `OAuth2PasswordBearer` + `get_current_active_user` already enforce per-route. |
| Token-validity check on app boot | Browser / Client | API / Backend | Browser fires `GET /users/me` (or `/api/auth/me` per D-17); backend signs/decodes. |
| Protected-route gating | Browser / Client | — | Render-time guard; reads `auth_token`. Backend's 401 is the second line of defence. |
| Session-expired redirect (401 → `/login`) | Browser / Client | — | `apiFetch` wrapper handles the redirect; backend just returns 401. |
| Provider list fetch (`GET /api/providers`) | API / Backend | — | Reads from `Settings.get_available_providers()`; needs auth. |
| Provider health probe (Ollama liveness, model presence) | API / Backend | — | Backend probes Ollama via `httpx` from `POST /api/chat/session`; browser never reaches Ollama directly (CORS + same-origin discipline). |
| Provider error rendering (F1–F4) | Browser / Client | API / Backend | Backend returns structured error body; frontend maps `error_code` → localized copy from UI-SPEC §F1–F4. |
| Logout | Browser / Client | — | Pure client action: clear `auth_token`, route to `/login`. No backend revocation in v1. |

**Why this matters:** the auth surface tempts misassignment in two specific ways. (a) the
provider liveness probe could be misplaced as a *frontend* probe (the browser hitting
Ollama directly); same-origin / CORS / firewall realities make this brittle, and the
backend already has `httpx` wired. (b) the boot-time token-validity check could be done
client-side by decoding the JWT to read `exp` — that's wrong because JWTs are signed by the
backend and the client has no business shipping the secret or verifying the signature; the
correct check is "ask the backend if this token still works."

### Pattern 1: Router setup + protected route (D-14, D-15, D-16)

**What:** `BrowserRouter` wraps the app at `main.tsx`; declarative `<Routes>` in `App.tsx`;
`<RequireAuth>` is a render-time guard that reads `localStorage` and redirects with
location state.

**When to use:** Every page that needs auth — currently just `/`, but the wrapper composes
once around `<ChatInterface />`.

**Example:**

```tsx
// frontend/src/App.tsx
```

```tsx
// frontend/src/components/auth/RequireAuth.tsx
import { Navigate, useLocation } from 'react-router-dom'
import type { ReactNode } from 'react'
import { getToken } from '../../lib/auth'

interface RequireAuthProps {
  children: ReactNode
}

export function RequireAuth({ children }: RequireAuthProps) {
  const location = useLocation()
  const token = getToken()

  if (!token) {
    // ?from= so post-login can redirect back
    const from = `${location.pathname}${location.search}`
    return <Navigate to={`/login?from=${encodeURIComponent(from)}`} replace />
  }

  return <>{children}</>
}
```

### Pattern 2: Auth-aware fetch wrapper (D-02, D-04)

**What:** A single `apiFetch()` function reads the token, injects the header, and handles
401-redirect centrally. Every `/api/*` call goes through it — `useChat`, `ProviderSelector`,
the boot-time `/users/me` check.

**When to use:** Any time the frontend hits `/api/*`. Direct `fetch('/api/...')` calls are
forbidden after this phase — they bypass the auth header and the 401 handler.

**Example:**

```ts
// frontend/src/lib/auth.ts
const TOKEN_KEY = 'auth_token'

export function getToken(): string | null {
  return localStorage.getItem(TOKEN_KEY)
}

export function setToken(token: string): void {
  localStorage.setItem(TOKEN_KEY, token)
}

export function clearToken(): void {
  localStorage.removeItem(TOKEN_KEY)
}

interface ApiFetchOptions extends Omit<RequestInit, 'headers'> {
  headers?: HeadersInit
}

export async function apiFetch(input: string, init: ApiFetchOptions = {}): Promise<Response> {
  const token = getToken()
  const headers = new Headers(init.headers)
  if (token && !headers.has('Authorization')) {
    headers.set('Authorization', `Bearer ${token}`)
  }
  const response = await fetch(input, { ...init, headers })

  if (response.status === 401) {
    clearToken()
    // Use full reload so React state is reset; ?flash carries the session-expired banner.
    const target = `/login?flash=session_expired&from=${encodeURIComponent(
      window.location.pathname + window.location.search
    )}`
    window.location.assign(target)
    // Throw so the caller's `.catch` runs and the in-flight UI doesn't continue.
    throw new Error('Session expired')
  }

  return response
}
```

**Why a window-level redirect (not `useNavigate`):** the wrapper must work outside
React-render context (called from `useChat` effects, from `ProviderSelector` mount, etc.).
A `window.location.assign` is the simplest option that doesn't require threading a router
hook into a non-component module. The full reload also clears any stale React state
(in-flight chat session, etc.) which is the desired behaviour per D-05.

**Auth context vs. plain functions (Discretion area):** A React context (`AuthContext`)
adds re-render machinery (`isAuthenticated`, `user`, etc.) that the codebase only needs in
two places: (a) showing the username in the header, (b) the `RequireAuth` gate. Both can
read `localStorage` directly. **Recommendation: skip `AuthContext` for this phase.** Use a
small `useCurrentUser()` hook that calls `apiFetch('/users/me')` once on mount. This keeps
the surface tiny and matches the existing codebase's "plain function" idiom (no other
contexts exist). If a later phase grows multiple auth-aware UI surfaces, lift to a context
then.

### Pattern 3: Login page submit flow (D-13, UI-SPEC §Right panel)

**What:** Form submit → `application/x-www-form-urlencoded` POST to `/token` → on 200,
`setToken` + redirect to `?from=` (or `/`); on 400, render Field.ErrorText.

**When to use:** Only on `/login`. The endpoint expects form-urlencoded (FastAPI
`OAuth2PasswordRequestForm`), not JSON.

**Example:**

```tsx
// frontend/src/components/auth/LoginForm.tsx (sketch)
import { Field, Input, Button } from '@chakra-ui/react'
import { useState, type FormEvent } from 'react'
import { useNavigate, useSearchParams } from 'react-router-dom'
import { setToken } from '../../lib/auth'

export function LoginForm() {
  const [error, setError] = useState<string | null>(null)
  const [loading, setLoading] = useState(false)
  const navigate = useNavigate()
  const [params] = useSearchParams()
  const from = params.get('from') ?? '/'

  const onSubmit = async (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault()
    setError(null)
    setLoading(true)
    const formData = new FormData(e.currentTarget)
    try {
      const response = await fetch('/token', {
        method: 'POST',
        body: new URLSearchParams({
          username: String(formData.get('username') ?? ''),
          password: String(formData.get('password') ?? ''),
        }),
      })
      if (!response.ok) {
        // 400 → "Incorrect username or password"
        setError('Username or password is incorrect.')
        return
      }
      const data = (await response.json()) as { access_token: string; token_type: string }
      setToken(data.access_token)
      navigate(from, { replace: true })
    } catch {
      setError('Could not reach the server. Try again.')
    } finally {
      setLoading(false)
    }
  }

  return (
    <form onSubmit={onSubmit}>
      <Field.Root invalid={!!error} mb="space.5">

```

**Critical detail:** `/token` is mounted at the **root** in `app/api/main.py` via
`app.include_router(auth.router)` — the path is `/token`, **not** `/api/token`. The Vite
dev proxy is configured for `/api/*` only (`vite.config.ts:9`). `/token` calls go directly
to the dev server's host, which means **either** the Vite proxy needs a second entry for
`/token`, **or** the auth router needs to be mounted under `/api/auth` so the existing
proxy catches it. The cleaner fix is the latter: mount auth at `/api/auth` and the login
endpoint becomes `POST /api/auth/token`. **This is a research finding the planner must
address** — the current code has `tokenUrl="token"` in `OAuth2PasswordBearer(tokenUrl="token")`
(`auth.py:23`), which is also wrong for the new mount path. Both must move together.

### Pattern 5: Provider health probe at session-create (D-07, D-18)

**What:** `POST /api/chat/session` calls a probe before allocating the session. The probe
returns either success (proceed to `chat_service.create_session()`) or a structured error
that maps 1:1 to the F1–F4 taxonomy.

**When to use:** Only at session-create time. **Do not** probe on every chat send (D-07
explicitly: "fire proactively at session-create time, not lazily on first chat send").

**Example:**

```python
# backend/app/services/provider_probe.py
import httpx
from typing import Literal
from pydantic import BaseModel
from app.config import settings


class ProbeError(BaseModel):
    """Structured probe failure — maps to UI F1/F2/F3."""

    error: Literal["ollama_unreachable", "model_not_installed", "missing_api_key"]
    message: str
    hint: str


_OLLAMA_PROBE_TIMEOUT_SECONDS = 1.5


async def probe_provider(provider: str, model: str) -> ProbeError | None:
    """Return None if the provider is usable, otherwise a structured ProbeError.

    Args:
        provider: One of "ollama", "openai", "anthropic".
        model: Model identifier.

    Returns:
        None on success, ProbeError on failure.
    """
    if provider == "ollama":
        return await _probe_ollama(model)
    if provider in {"openai", "anthropic"}:
        return _probe_cloud(provider)
    # Unknown provider — let the existing 400 path in routes.py handle it.
    return None


async def _probe_ollama(model: str) -> ProbeError | None:
    url = f"{settings.ollama_base_url.rstrip('/')}/api/tags"
    try:
        async with httpx.AsyncClient(timeout=_OLLAMA_PROBE_TIMEOUT_SECONDS) as client:
            response = await client.get(url)
            response.raise_for_status()
    except (httpx.ConnectError, httpx.TimeoutException, httpx.HTTPStatusError):
        return ProbeError(
            error="ollama_unreachable",
            message=f"Can't reach Ollama at {settings.ollama_base_url}.",
            hint="Run `ollama serve` and retry, or pick another provider.",
        )

    payload = response.json()
    # Ollama /api/tags returns: { "models": [ { "name": "...", "model": "...", ... }, ... ] }
    # Match on `name` AND `model` to be robust to minor shape drift.
    available = {
        entry.get("name") for entry in payload.get("models", [])
    } | {entry.get("model") for entry in payload.get("models", [])}
    if model not in available:
        return ProbeError(
            error="model_not_installed",
            message=f"Ollama is running but {model} isn't installed.",
            hint=f"Run `ollama pull {model}` or pick a different model.",
        )
    return None


def _probe_cloud(provider: str) -> ProbeError | None:
    # v1: env-var presence only. Phase 4.5 ships live key validation.
    key = getattr(settings, f"{provider}_api_key", None)
    if not key:
        env_var = f"{provider.upper()}_API_KEY"
        return ProbeError(
            error="missing_api_key",
            message=f"{provider.title()} needs an API key.",
            hint=f"Set {env_var} on the backend and restart.",
        )
    return None
```

```python
# backend/app/api/routes/routes.py — modified create_session
@router.post("/api/chat/session", status_code=status.HTTP_201_CREATED)
async def create_session(
    chat_service: ChatService = Depends(get_chat_service),
```

local. The 1.5 s timeout caps the worst case. CONTEXT.md D-07 calls out "~200 ms latency
on selection. Acceptable."

**FastAPI HTTPException with a dict detail:** FastAPI serializes `detail` directly into the
response body's `detail` key. The frontend reads `response.json().detail` to get

`routes.py:163`).

### Pattern 6: Frontend error mapping (UI-SPEC §F1–F4)

**What:** A small mapping module turns `error` codes into UI-SPEC-locked copy. Keeps the
copy out of components.

**Example:**

```ts
// frontend/src/lib/providerErrors.ts
export type ProviderErrorCode =
  | 'ollama_unreachable'
  | 'model_not_installed'
  | 'missing_api_key'
  | 'providers_fetch_failed'

export interface ProviderErrorView {
  code: ProviderErrorCode
  message: string
  hint: string
  // Inline code spans the banner should render in mono font (UI-SPEC §Selector error banner)
  inlineCode: string[]
}

export interface BackendProbeError {
  error: ProviderErrorCode
  message: string
  hint: string
}

export function mapProbeError(
  body: BackendProbeError,
  context: { provider: string; model: string; ollamaBaseUrl?: string }
): ProviderErrorView {
  switch (body.error) {
    case 'ollama_unreachable':
      return {
        code: body.error,
        message: body.message,
        hint: body.hint,
        inlineCode: ['ollama serve'],
      }
    case 'model_not_installed':
      return {
        code: body.error,
        message: body.message,
        hint: body.hint,
        inlineCode: [context.model, `ollama pull ${context.model}`],
      }
    case 'missing_api_key':
      return {
        code: body.error,
        message: body.message,
        hint: body.hint,
        inlineCode: [`${context.provider.toUpperCase()}_API_KEY`],
      }
    default:
      return { code: 'providers_fetch_failed', message: body.message, hint: body.hint, inlineCode: [] }
  }
}

export const PROVIDERS_FETCH_FAILED: ProviderErrorView = {
  code: 'providers_fetch_failed',
  message: "Couldn't load providers.",
  hint: 'Check that the backend is running.',
  inlineCode: [],
}
```

The `<SelectorErrorBanner>` consumes a `ProviderErrorView` and renders the locked UI-SPEC
visual contract (icon, copy, retry button). The retry handler comes from the parent —
either re-call `apiFetch('/api/providers')` or re-call session-create.

### Anti-Patterns to Avoid

- **Direct `fetch('/api/...')` calls anywhere in `frontend/src/`.** They bypass the auth
  header and the 401 redirect. After this phase, every call goes through `apiFetch`. Add an
  ESLint rule (or at minimum a code-review check) to enforce.
- **Decoding the JWT client-side to read `exp` and check expiry.** The client doesn't have
  the secret; even if it did, server clock skew matters. Always ask the backend (`GET
  /users/me`) — a 401 is the source of truth.
- **Probing Ollama from the browser.** Same-origin policy + Ollama's CORS defaults
  (closed) + Phase 5 compose (where Ollama is on `host.docker.internal`) all conspire
  against this. Probe from the backend.
- **Returning a generic 500 from the probe failure path.** The frontend cannot distinguish
  F1/F2/F3 from a generic 500 — use 502 (ollama_unreachable) and 400
  (model_not_installed / missing_api_key) with a structured `detail`.
- **Storing the password in component state any longer than the submit handler.** The
  password lives only in the `<input>`'s DOM value and the `FormData` snapshot. No
  `setPassword` state — it's not needed.
- **Mixing the `?flash=session_expired` query param with React Router state.** D-02 and
  UI-SPEC §Session-expired flash both use `?flash=` (a query param), not router state. The
  flash banner reads `useSearchParams()` and dismisses by `replaceState` after first
  render.
- **Calling `useNavigate()` from `lib/auth.ts`.** Hooks only run inside React components.
  The 401 handler must use `window.location.assign()` (or, if truly needed, accept a
  navigate callback as a parameter — but that ties the wrapper to React's render cycle and
  is heavier than necessary).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Client-side routing + protected routes | A custom `useState`-driven view switcher in `App.tsx` | `react-router-dom` v7 declarative mode | `<Navigate>`, `useLocation`, `useNavigate`, `useSearchParams` cover every case here. Locked by D-14. |
| JWT decode/validate on the client | Custom base64-decode + clock comparison | Just call `GET /users/me` — let the backend tell you if the token works | Client-side decode requires the secret (you don't have it) or unsigned trust (insecure). Ask the source of truth. |
| Password show/hide toggle | Custom button + state + manual `type` swap | Same pattern but use lucide `Eye` / `EyeOff` icons + `aria-pressed` | Trivial code; the value is in the icon set + ARIA discipline (UI-SPEC §Right panel item 4). |
| Form validation | Hand-rolled "is this empty" checks | HTML5 `required` + Chakra `Field.Root invalid={...}` | UI-SPEC §Form validation says no client-side rules beyond `required`; backend enforces. |
| SSE parsing | Custom buffer-and-newline parser | The existing `lib/parseSSE.ts` (already shipped) | Already battle-tested. Don't touch. |
| HTTP retries on the probe | Manual `for attempt in range(3)` loop | Don't retry — fail fast | The probe is a UX latency budget (~200 ms). Retrying multiplies the worst-case wait. F1 + Retry button is the user's retry path. |
| Color/typography system from scratch | Hardcoded hex codes throughout components | UI-SPEC §Design System tokens via Chakra `createSystem` | UI-SPEC locks 12 OKLCH tokens. Theme system file at `frontend/src/theme/index.ts` is the single source. |
| Ollama tag parsing | Custom regex against the response text | `httpx` JSON + dict access on `payload["models"][i]["name"]` and `["model"]` | `/api/tags` is documented JSON. [CITED: github.com/ollama/ollama/blob/main/docs/api.md] |
| Session-expired flash | Custom prop drilling from `apiFetch` to `<Login>` | `?flash=session_expired` query param | Survives the full-reload that `window.location.assign` triggers. Read once, then `replaceState` to clear. |

**Key insight:** Phase 4.2 is a *wiring* phase. Almost nothing is novel — it's three
existing pieces (the JWT backend, the Chakra v3 component layer, the React frontend)
connected by two thin glue modules (`lib/auth.ts`, `services/provider_probe.py`). The
biggest mistake the planner could make is to introduce abstractions (auth context, hook
hierarchies, custom error frameworks) that this scope doesn't support. Lean on what's
shipped.

## Common Pitfalls

### Pitfall 1: `/token` is mounted at the root, not under `/api/*`

**What goes wrong:** `LoginForm` posts to `/token`, but the Vite dev proxy only forwards
`/api/*` to the backend. Without a proxy entry for `/token`, the dev server returns a 404
HTML page and the form silently breaks. In production (no Vite proxy) it depends on how
the deployment routes the auth surface.

**Why it happens:** `auth.router` is included with no prefix in `main.py` (`app.include_router(auth.router)`),
so `/token` and `/users/me` are at the root. `vite.config.ts` only proxies `/api/*`.

**How to avoid:** Move auth under `/api/auth` — change `app.include_router(auth.router,
prefix="/api/auth")` in `main.py`, update `OAuth2PasswordBearer(tokenUrl="api/auth/token")`
in `auth.py:23`, update the existing integration test that posts to `/token` to post to
`/api/auth/token` instead, and update the existing `/users/me` test (if present). The
frontend then targets `/api/auth/token` and the existing proxy catches it. **This is a
breaking rename for the test suite — call it out explicitly in the plan.**

**Warning signs:** Login button "does nothing" in dev; Network tab shows a 404 to
`http://localhost:5173/token` instead of `http://localhost:8000/token`.

### Pitfall 2: localStorage key collision and cross-tab confusion

**What goes wrong:** The test in `useChat.test.ts:33-46` defines its own localStorage mock.
A naïve `lib/auth.ts` test that doesn't replicate the mock will fail. Cross-tab logout is
explicitly out of scope (CONTEXT.md "Deferred Ideas") but adjacent: tab A logs out, tab B
keeps the stale token until its next 401, which then redirects.

**Why it happens:** `localStorage` is a global side effect; jsdom's default behaviour
varies. The existing test already wires its own mock — frontend tests in this phase should
follow the same pattern.

**How to avoid:** Reuse the localStorage mock pattern from `useChat.test.ts`. Document the
cross-tab limitation as a known gap (it already is in CONTEXT.md). Do not add a `storage`
event listener in v1.

### Pitfall 3: httpx timeout discipline on the probe

**What goes wrong:** `httpx.AsyncClient()` with no `timeout` argument waits forever (well —
5 minutes by default, which is forever for a UX latency budget). A user picks Ollama, it's
not running, and the session-create call hangs. The frontend spinner spins; the user
thinks the app is broken.

**Why it happens:** `httpx`'s default timeout is 5 s for connect + 5 s for read. That's too
long for a "is Ollama up" check, and the failure surface is a hang, not a 502.

**How to avoid:** Always pass `timeout=` (or a `httpx.Timeout()` object) to `AsyncClient`.
1.5 s is a reasonable budget for a localhost probe. Catch `httpx.ConnectError`,
`httpx.TimeoutException`, and `httpx.HTTPStatusError` (from `raise_for_status`) all as
`ollama_unreachable`. Match the response code on **502 Bad Gateway** (the upstream is
unreachable) — not 500 (server-side error) and not 503 (this server unavailable).

**Warning signs:** Backend logs show stuck `httpx` operations; frontend shows the spinner
for >2 s on a misconfigured Ollama.

### Pitfall 4: Probe error code → HTTP status mapping

**What goes wrong:** Returning 400 for *every* probe failure makes it harder for the
frontend to distinguish "we screwed up the request" from "the upstream is down" from "the
server is misconfigured." Returning 500 for everything is worse — the frontend can't show
a useful banner.

**Why it happens:** It's tempting to use 400 as a generic "this probe failed" code.

**How to avoid:** Use the proposed mapping:

| Probe error | HTTP | Reason |
|-------------|------|--------|
| `ollama_unreachable` | **502** | Upstream gateway unreachable. |
| `model_not_installed` | **400** | The client picked a model the server can't satisfy — request is invalid. |
| `missing_api_key` | **400** | Same shape: server can't fulfil the requested provider. |

The **frontend** doesn't actually need to distinguish 502 from 400 — it reads
`response.json().detail.error` either way. But the status code is observable from logs and
helps debugging. Document the mapping; don't invent variations.

### Pitfall 5: Chakra v3 `<Select.Root>` collection requirement vs. NativeSelect simplicity

**What goes wrong:** UI-SPEC §"Selector dropdowns" calls for the custom `Select.Root` /
`Select.Trigger` / `Select.Content`. v3 docs require `useListCollection({ items: ... })` to
manage the option list — that's a hook call inside the component, plus a portal
positioner, plus item rendering. For two short lists (3–4 providers, 4–5 models per
provider) this is heavier than warranted.

**Why it happens:** The custom Select gives editorial styling control (UI-SPEC). The
existing `<select>` is a styled HTML element that doesn't compose cleanly with Chakra
tokens.

**How to avoid:** Two options. (a) Adopt `Select.Root` per UI-SPEC and accept the
collection plumbing — the right answer if the visual contract demands it. (b) Use
`NativeSelect.Root` + `NativeSelect.Field` with `<option>` children — much lighter, picks
up Chakra style props, may not match the visual contract exactly. **Recommend (b) and ask
the planner to confirm with the UI-SPEC owner** whether NativeSelect satisfies §"Selector
dropdowns" — the visual diff is mostly the dropdown portal animation.

### Pitfall 6: `_metadata` access pattern for session resolution

**What goes wrong:** `routes.py:134` reads `chat_service._metadata[session_id]` directly.
After Phase 5 swaps `_metadata` for a Postgres-backed store, this access pattern breaks.
Adding probe logic *next to* this access tempts the planner to extend the same pattern.

**Why it happens:** `_metadata` is a private dict the routes layer reaches into. It works
today; it won't work after Phase 5.

**How to avoid:** This phase **should not** touch the `_metadata` access pattern (that's a
Phase 5 problem). But the new probe code should not introduce *new* private-dict access.
Probe inputs are `request.provider` and `request.model` — both are public. Probe outputs
are returned via `HTTPException`. No private access needed.

### Pitfall 7: Frontend session-expired flash flicker

**What goes wrong:** `apiFetch` redirects to `/login?flash=session_expired&from=…`. The
`<Login>` component reads `searchParams.get('flash')` on first render and shows the
banner. On the *next* render (or after the user types in the form), the param is still in
the URL — if the user submits and gets a 400, the banner re-shows alongside the new error.

**Why it happens:** The query param persists until something removes it.

**How to avoid:** After reading `flash` on mount, call `setSearchParams` with an empty
object (or `replace`) to clear it. Use `useEffect` so the read happens once. UI-SPEC
§Right panel item 2 already specifies "Auto-dismiss on user input (clear from location
state via `replaceState`)" — implement exactly that.

### Boot-time `/users/me` check (D-15 + D-17)

```tsx
// frontend/src/components/auth/RequireAuth.tsx — extended sketch with verify
import { Navigate, useLocation } from 'react-router-dom'
import { useEffect, useState, type ReactNode } from 'react'
import { apiFetch, getToken } from '../../lib/auth'

interface User { username: string }

export function RequireAuth({ children }: { children: ReactNode }) {
  const location = useLocation()
  const token = getToken()
  const [verified, setVerified] = useState<'pending' | 'ok' | 'fail'>(token ? 'pending' : 'fail')

  useEffect(() => {
    if (!token) return
    let cancelled = false
    apiFetch('/users/me')                          // or '/api/auth/me' per D-17 resolution
      .then((r) => (r.ok ? r.json() as Promise<User> : Promise.reject()))
      .then(() => { if (!cancelled) setVerified('ok') })
      .catch(() => {
        // apiFetch already cleared token + redirected on 401; this is a network error.
        if (!cancelled) setVerified('fail')
      })
    return () => { cancelled = true }
  }, [token])

  if (verified === 'fail') {
    const from = `${location.pathname}${location.search}`
    return <Navigate to={`/login?from=${encodeURIComponent(from)}`} replace />
  }
  if (verified === 'pending') return null  // or a tiny spinner
  return <>{children}</>
}
```

## Open Questions (RESOLVED)

1. **D-17 reuse vs. new endpoint?**
   - What we know: `GET /users/me` already exists at `auth.py:195`; D-17 spec says "new
     `/api/auth/me`."
   - What's unclear: whether the planner should rename the existing route or add a second
     route alongside it.
   - Recommendation: rename to `/api/auth/me` as part of the auth-prefix move (Pitfall 1),
     and update the existing test. One canonical path; no aliasing.
   - **RESOLVED (Plan 02):** auth router mounted at `/api/auth`; `/users/me` renamed to
     `GET /api/auth/me`. Integration stub in `test_auth_flow.py` tests the new path.

2. **Custom Chakra `Select.Root` vs. `NativeSelect`?**
   - What we know: UI-SPEC §"Selector dropdowns" specifies `Select.Root` styling.
   - What's unclear: whether NativeSelect with the same Chakra style props satisfies the
     visual contract.
   - Recommendation: implement with NativeSelect first (lighter, faster, matches existing
     `ProviderSelector.tsx`'s mental model). Re-render test: if the dropdown looks "templatey"
     vs. UI-SPEC, lift to `Select.Root` in a follow-up.
   - **RESOLVED (Plan 06):** NativeSelect adopted per recommendation; visual contract
     reviewed against UI-SPEC §"Selector dropdowns" at checkpoint in Plan 06 Task 2.

3. **Does `apiFetch` need a "skip-401-redirect" escape hatch?**
   - What we know: the `/users/me` boot check legitimately tolerates a 401 (it's how we
     detect a stale token).
   - What's unclear: whether routing the 401 through `apiFetch`'s redirect is acceptable
     for that one call (effectively making the boot check trigger the redirect we want
     anyway).
   - Recommendation: the redirect IS what we want on a stale token, so route the
     `/users/me` call through `apiFetch` and let the wrapper handle it. No escape hatch
     needed.
   - **RESOLVED (Plan 03):** `apiFetch` has no escape hatch; `RequireAuth` routes the
     `/api/auth/me` call through `apiFetch` so a 401 triggers the redirect automatically.

4. **`ollama_base_url` exposure to the frontend for the F1 banner?**
   - What we know: F1 copy includes "Can't reach Ollama at {OLLAMA_BASE_URL}." UI-SPEC
     §F1 implies the URL is rendered.
   - What's unclear: how the frontend learns the URL — env-injected at build time, or
     surfaced via the probe error body?
   - Recommendation: the probe's `message` field already includes the URL ("Can't reach
     Ollama at http://localhost:11434"). Frontend renders `body.detail.message` directly.
     No extra channel needed.
   - **RESOLVED (Plan 02 + Plan 06):** URL is embedded in `ProbeError.message` by the
     backend; frontend renders `body.detail.message` from the probe response — no extra
     channel needed.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | `pwdlib[argon2]` for password hashing (already in use); `pyjwt` HS256 for token signing (already in use); `OAuth2PasswordBearer` flow (already in use). No password complexity rules in v1 (UI-SPEC §Form validation). |
| V3 Session Management | yes | JWT with `exp` claim, 60-min default (`config.py:25`). No refresh tokens, no revocation list — both are deferred. Session-expired UX is the redirect + flash. |
| V4 Access Control | yes | `Depends(get_current_active_user)` on every `/api/*` route except `/health`. Frontend `<RequireAuth>` is convenience; backend is the authority. |
| V5 Input Validation | yes | `OAuth2PasswordRequestForm` (FastAPI built-in) validates the form payload. `SessionCreateRequest` Pydantic model + `ProbeError` Pydantic model validate the new probe surface. |

| V7 Error Handling | yes | Structured 400/401/502 responses; error messages don't leak stack traces. The probe's `hint` field is user-facing and includes shell commands — verify they're safe to render as plain text + Chakra `<code>` (no `dangerouslySetInnerHTML`). |
| V8 Data Protection | yes | localStorage XSS exposure documented and accepted (D-01). No PII flows through the new surfaces. |

### Known Threat Patterns for {React + FastAPI + JWT-in-localStorage}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS exfiltrating the JWT from localStorage | Information disclosure | Phase 8 ships CSP + `rehype-sanitize`. v1 acknowledges and documents the risk per CONTEXT.md D-01. **No new XSS surface introduced by this phase** — the login form renders strings, not HTML; the error banner renders fixed copy + inline `<code>` (text-only). |
| CSRF on `POST /token` | Tampering | The token endpoint is intentionally public; CSRF doesn't apply because there's no ambient cookie. Form posts are origin-controlled by CORS. |
| Brute-force login | Denial of Service | Out of scope — ADR-009 explicitly drops rate limiting from v1. Documented limitation. |
| JWT replay after logout | Spoofing | Out of scope — no revocation list in v1 (deferred to v2 / Phase 5 PG-backed sessions). Mitigated by 60-min `exp`. |
| Session fixation via the `?from=` redirect | Spoofing | The `from` parameter is `URIComponent`-encoded and passed to `useNavigate` which only routes within the SPA — no open-redirect surface because react-router doesn't accept absolute URLs from `from`. **Verify** by adding a test that asserts a `from=https://evil.example.com` redirect lands on `/` (or stays on `/login`), not the external URL. |
| Path traversal / SSRF via the probe URL | Information disclosure | The probe URL is built from `settings.ollama_base_url` (server-controlled), not user input. No SSRF. |
| Information leak via probe error messages | Information disclosure | F1–F4 messages disclose `OLLAMA_BASE_URL` and env-var names — both are configuration, not secrets. Acceptable. **Do not** include the bound port if it's a non-default ephemeral port (it isn't here). |
| Open-redirect via `?flash=` | — | The `flash` param is read once and matched against a known-string allowlist (`session_expired`). Don't render arbitrary flash text. |
| Argon2 hashing wall-clock cost | Denial of Service | Each login does one Argon2 verify (~50–100 ms with default params). At ADR-009 "no rate limiting" + small `AUTH_USERS` size, this is acceptable; not a meaningful DoS vector. |

**Phase 4.2 introduces NO new attack surface** beyond what `REQ-auth-backend` already
shipped. The frontend changes consume the existing endpoints; the backend addition is one
new probe (no auth bypass, no SQL, no user input rendering as HTML). The session-create
probe sends an HTTP GET to a server-configured URL (`OLLAMA_BASE_URL`) — no user input
flows into the URL.
