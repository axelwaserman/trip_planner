# Phase 04.2 Learnings: unbreak-the-app

## Decisions

### Mount auth router at /api/auth (not at root)
The auth router prefix moved from root â†’ `/api/auth` so the existing Vite `/api/*` proxy reaches it without a second proxy entry. `/users/me` was renamed to `/me` (resolves to `/api/auth/me` once the prefix applies).

**Rationale:** RESEARCH.md Pitfall 1 â€” the bare `/token` and `/users/me` paths could never be reached from the SPA without per-route proxy edits. One canonical mount removes the divergence.
**Source:** 04.2-02-SUMMARY.md, 04.2-RESEARCH.md (Pitfall 1)

---

### Probe at session-create, not lazily on first chat send
`probe_provider()` runs synchronously inside `POST /api/chat/session` *before* `chat_service.create_session()`. Failing probes raise `HTTPException(detail=probe.model_dump())` and never allocate a session.

**Rationale:** D-07 + D-18 (CONTEXT.md) â€” proactive probing surfaces F1-F4 at the point the user picked the provider, not several turns later mid-conversation.
**Source:** 04.2-04-be-session-probe-wire-PLAN.md, 04.2-04-SUMMARY.md

---

### 502 for connectivity, 400 for misconfig
`provider_unreachable` (originally `ollama_unreachable`) â†’ 502 Bad Gateway; `model_not_installed` and `missing_api_key` â†’ 400 Bad Request.

**Rationale:** RESEARCH.md Pitfall 4 â€” connectivity is upstream gateway failure semantics; model/key issues are client-side misconfig the user can fix without leaving the page.
**Source:** 04.2-04-SUMMARY.md, 04.2-RESEARCH.md (Pitfall 4)

---

### JWT in localStorage under key auth_token
`getToken/setToken/clearToken` back the JWT with `localStorage.getItem('auth_token')`.

**Rationale:** D-01 explicitly accepts localStorage for v1; httpOnly cookies + CSRF + refresh tokens are deferred to Phase 8.
**Source:** 04.2-CONTEXT.md (D-01), 04.2-03-SUMMARY.md

---

### apiFetch on 401 â€” clear, redirect, throw
On 401, `apiFetch` clears the token, navigates to `/login?flash=session_expired&from=<path>` via `window.location.assign`, and throws `Error('Session expired')` so callers' `.catch()` runs.

**Rationale:** D-02 + D-04. Window-level redirect (not `useNavigate`) so the wrapper works outside React context (boot-time `/api/auth/me` probe, useChat effects, ProviderSelector mount).
**Source:** 04.2-CONTEXT.md (D-02, D-04), 04.2-03-SUMMARY.md, frontend/src/lib/auth.ts

---

### LoginForm uses bare fetch, not apiFetch
`POST /api/auth/token` is called via `fetch()`, not `apiFetch()`. No token exists at login time, so apiFetch's Authorization header injection would attach a stale value and its 401 handler would fire a redirect loop into `/login`.

**Rationale:** D-05 contract.
**Source:** 04.2-CONTEXT.md (D-05), 04.2-05-fe-login-and-routing-PLAN.md

---

### Custom Chakra system with 11 OKLCH tokens
Replaced `defaultSystem` with `createSystem(defaultConfig, defineConfig({...}))` carrying the UI-SPEC palette (bg/fg/border/accent/danger), Fraunces+Inter fonts, and a small radii scale.

**Rationale:** UI-SPEC Â§"Color Palette" + Â§"Typography" lock OKLCH tokens for parity across light/dark surfaces; defaults wouldn't render the editorial Surface 1 hero correctly.
**Source:** 04.2-03-SUMMARY.md, 04.2-UI-SPEC.md

---

### Chat lives at /app, not /
After UAT, the chat route moved from `/` to `/app`; `/` and `*` redirect to `/app`. RequireAuth and apiFetch's 401 handler also omit the `?from=` query param when the source is the default landing.

**Rationale:** Manual UAT finding â€” the post-login `/?from=%2F` URL was noisy, and the user wanted a stable `/app` landing for future routes.
**Source:** Commit fdd61a9 (`fix(04.2): UAT fixes`), frontend/src/App.tsx

---

### Drop the available=False short-circuit; probe owns missing-key
`POST /api/chat/session` no longer rejects providers with `available=False` before the probe runs. The probe is the single authority on the F1-F4 taxonomy.

**Rationale:** UAT finding â€” F3 (missing OpenAI API key) was unreachable because the route's older `available=False` 400 (string detail body) fired first, hiding the structured `missing_api_key` ProbeError the frontend banner needs.
**Source:** Commit fdd61a9, backend/app/api/routes/routes.py

---

### Render every backend-exposed provider in the dropdown
ProviderSelector no longer filters on `available`; the user can pick OpenAI/Anthropic even with no key, and the probe surfaces the F3 banner.

**Rationale:** Same UAT finding â€” hiding providers made F3 untestable from the UI. Now the user sees the option, picks it, and the banner explains exactly which env var to set.
**Source:** Commit fdd61a9, frontend/src/components/ProviderSelector.tsx

---

### Wire-level error code is provider_unreachable, not ollama_unreachable
Renamed the F1 code so the symbol generalizes to future cloud-provider connectivity failures (DNS/timeout/4xx/5xx). The Ollama-specific error message body is preserved, but the routing tag is generic.

**Rationale:** PR #7 review feedback â€” `ollama_unreachable` is a wire-level field, not a copy field. The 502 boundary must key on a generic concept so future cloud probes plug in without a route change.
**Source:** Commits 4e9f997, a261fca, d12a322 (PR #7 fixups)

---

### Cross-module taxonomies use StrEnum, not duplicated Literal unions
`ProbeErrorCode` (StrEnum) is the single source of truth, imported by both `provider_probe.ProbeError.error` and `models.SessionCreateError.error`. CLAUDE.md now documents this as the project convention.

**Rationale:** PR #7 review â€” keeping two parallel `Literal["a", "b", "c"]` unions in sync across modules is a DRY violation that breaks silently when only one side is updated.
**Source:** Commit 4e9f997, CLAUDE.md "Key constraints" section

## Lessons

### Backgrounded subagents stall at the 600s SSE watchdog on long plans
Both Wave 1 plans (02 and 03) hit Claude Code's stream-idle-timeout mid-execution, leaving 1 commit each on a worktree branch + uncommitted partial edits. The orchestrator merged the worktree branches, salvaged the uncommitted edits, and finished inline.

**Context:** Long-running plans with multiple file writes + heavy tool calls accumulate enough cache_read context that the SSE layer gives up between a tool_result and the next assistant turn. Heartbeat checkpoints (#2410-class) help but don't fully prevent it on 200K+ contexts.
**Source:** 04.2-02-SUMMARY.md (Deviations), 04.2-03-SUMMARY.md (Deviations)

---

### vitest 4 + jsdom 29 ship without a usable localStorage
Tests assuming `localStorage.clear()` works failed with `TypeError: localStorage.clear is not a function` and an unconfigured `--localstorage-file` warning. Fix: install a small in-memory Storage shim in `frontend/src/test-setup.ts` so all tests get a consistent Storage API.

**Context:** Existing `useChat.test.ts` had inlined its own per-test localStorage mock; new tests for the auth helper expected the jsdom default to "just work."
**Source:** 04.2-03-SUMMARY.md (Deviations), frontend/src/test-setup.ts

---

### Pydantic Settings instances reject arbitrary setattr â€” monkeypatch via the module reference
Trying `monkeypatch.setattr(settings, "get_available_providers", lambda: ...)` raises `ValueError: "Settings" object has no field "get_available_providers"` because Settings forbids unknown fields. Workaround: replace `routes.settings` and `provider_probe.settings` with a `FakeSettings` instance.

**Context:** Plan 04 integration test for `missing_api_key` needed to bypass the route's older `available=False` check; only by patching the *module attribute* (not the Settings instance) can the test substitute a custom shape.
**Source:** 04.2-04-SUMMARY.md (Deviations)

---

### Chakra v3 Box-as-form narrows onSubmit to HTMLDivElement
`<Box as="form" onSubmit={(e: FormEvent<HTMLFormElement>) => ...}>` fails the type check: Chakra's polymorphic Box types narrow `onSubmit` to `FormEvent<HTMLDivElement>`. Use a real `<form>` element.

**Context:** Plan 05 originally used `Box as="form"` per the plan; tsc caught it during build.
**Source:** 04.2-05-SUMMARY.md (Deviations)

---

### Wave-0 test stubs only catch what the planner thought of
The Wave 0 stub for the F3 (missing API key) integration test passed against the route's older `available=False` short-circuit (string detail body), masking the fact that the structured `missing_api_key` ProbeError was unreachable from the UI. Manual UAT â€” not the test suite â€” exposed it.

**Context:** Stub-driven gates are necessary but not sufficient. They catch presence/shape regressions, not user-flow ergonomics.
**Source:** Commit fdd61a9, 04.2-CONTEXT.md (UAT findings preserved in commit message)

---

### `bash` `cd ./frontend` mid-session drifts the orchestrator's cwd
Running `npm install` from a Bash tool call without preserving cwd shifted the orchestrator's working directory into a worktree subdirectory; the next `git` command operated on the wrong tree, including a `git stash` that almost stranded an uncommitted file. Recovery: explicit `cd /Users/axel/code/trip_planner && pwd` re-anchor and absolute `git -C` for worktree operations.

**Context:** The `worktree-path-safety.md` reference exists for exactly this; the orchestrator skipped it on a one-off install and got bitten.
**Source:** Session log (uncaptured in artifacts); referenced in PR #7 description

---

### Test setup leaks across files when a global shim is added
The localStorage shim in test-setup.ts plus the inline mock in useChat.test.ts coexist fine, but only because the inline mock uses `Object.defineProperty(globalThis, 'localStorage', ...)` to override the shim per test. A bare `localStorage.clear()` in a new test file relies entirely on the shim â€” meaning a future test that defines its own mock at module top-level may shadow the shim and break sibling tests.

**Context:** vitest does not isolate global state per file by default.
**Source:** frontend/src/test-setup.ts, frontend/src/hooks/__tests__/useChat.test.ts

---

### CI runs `ruff format --check`, not just `ruff check`
Local pre-push gate ran `uv run ruff check` and passed. CI ran `uv run ruff format --check` and failed on a 3-line `httpx.AsyncClient(...)` block that fit on one line. Required a fixup commit + second autosquash + second force-push.

**Context:** Local convenience commands diverge from CI invocations. Always read `.github/workflows/ci.yml` before claiming a clean push.
**Source:** Session log; PR #7 CI run history

---

### --autosquash drops --no-ff merge commits silently
The 31-commit branch had two `chore: merge wave1 plan-XX partial` `--no-ff` commits from the worktree-stall recovery. After `git rebase -i --autosquash origin/master`, those merge commits vanished â€” their content was preserved in the linear ancestry, but the commit messages were gone.

**Context:** This is correct git behavior (`--rebase-merges` would preserve them) but unexpected if the merge commits documented something.
**Source:** Session log; pre/post-rebase commit count went 31 â†’ 29

## Patterns

### Wave-0 test-stub gate
Plan 01 ships entirely test code (xfail + test.todo) that fails-but-doesn't-error. Each stub names the downstream plan that must turn it green. Subsequent plans cannot ship without removing the marker, which forces an explicit "I confirmed the gate passed" step.

**When to use:** Multi-wave phases where test files for not-yet-existing modules would otherwise have to be written *after* the implementation, losing the contract-first benefit. The pattern is most valuable when the contract is locked (UI-SPEC, RESEARCH.md) and the implementation order is decoupled.
**Source:** 04.2-01-SUMMARY.md (patterns-established)

---

### Conditional import shield for not-yet-existing modules
Backend test stubs that need a not-yet-existing module wrap the import in `try/except ImportError` with a `probe_provider = None` fallback so pytest collection doesn't crash. The per-test xfail decorator + `assert probe_provider is not None` gate fails fast under xfail until the module lands.

**When to use:** Wave-0 stub plans where the test file must exist (for collection in CI) before its target module exists.
**Source:** 04.2-01-SUMMARY.md, backend/tests/unit/test_provider_probe.py

---

### Structured probe error envelope
`ProbeError(BaseModel)` carries `error: ProbeErrorCode`, `message: str`, `hint: str`. `HTTPException(detail=probe.model_dump())` surfaces the structured envelope to the frontend; `mapProbeError(detail, {provider, model})` converts it into a `ProviderErrorView` for the banner.

**When to use:** Anywhere the backend needs to communicate a misconfiguration the user can fix in their own environment (local daemon down, missing env var, model not installed). The structured shape lets the frontend render hint copy + inline shell snippets without the backend having to know about UI components.
**Source:** 04.2-RESEARCH.md (Pattern 5), 04.2-04-SUMMARY.md, frontend/src/lib/providerErrors.ts

---

### apiFetch wrapper as the single auth boundary
A single `apiFetch(input, init)` function injects `Authorization: Bearer <jwt>` and handles 401 â†’ clear-and-redirect for every `/api/*` call. Components never call `fetch()` directly except for the login POST (which has no token yet).

**When to use:** SPAs with bearer-token auth where the alternative is sprinkling `Authorization` header logic and 401 handling across every fetch call site. The wrapper is the only place that owns the redirect contract.
**Source:** 04.2-CONTEXT.md (D-04), 04.2-03-SUMMARY.md, frontend/src/lib/auth.ts

---

### RequireAuth as a render-time guard, not a router config
`<RequireAuth>` is a component that wraps the protected element inside `<Route element={...}>`. It calls `apiFetch('/api/auth/me')` on mount, renders `null` while pending, redirects with `?from=` on fail, renders children on success. No router-level beforeEnter or loader function.

**When to use:** React Router declarative mode where the auth check needs to fire on every route entry (including direct URL paste / refresh) but you want to keep the route tree as plain JSX without `loader:` functions.
**Source:** 04.2-RESEARCH.md (Pattern 1), 04.2-05-SUMMARY.md, frontend/src/components/auth/RequireAuth.tsx

---

### Tunable thresholds live on Settings, not as module constants
Probe timeouts, retry counts, expiry windows, etc. go on the `Settings` BaseSettings class so they can be overridden per environment via env vars. Module constants are reserved for *contract* values (JSON keys, MIME types, status codes).

**When to use:** Anywhere the value is a tuning knob rather than a wire-level invariant. The PR review on Phase 4.2 promoted `_OLLAMA_PROBE_TIMEOUT_SECONDS` to `Settings.provider_probe_timeout_seconds`.
**Source:** PR #7 review, CLAUDE.md "Key constraints" section

---

### StrEnum for cross-module string taxonomies
When a fixed set of stable string codes appears in two or more places (e.g. `ProbeError.error` and `SessionCreateError.error`), define it once as a `StrEnum` and import it. Pydantic accepts StrEnum for `Literal`-style fields and serializes them as strings.

**When to use:** Any time `Literal["a", "b", "c"]` would otherwise need to be repeated across modules. The wire format is preserved (StrEnum members serialize to their `.value` strings) and updates land in one place.
**Source:** PR #7 review, CLAUDE.md "Key constraints" section, backend/app/services/provider_probe.py

## Surprises

### Probe-test for missing_api_key needed the route refactor to be testable
The Wave 0 integration stub for F3 was written assuming the probe would own the missing-key signal. When Plan 04 wired the probe in, the existing `if not providers[request.provider]["available"]` short-circuit fired first, masking the structured ProbeError. Plan 04 worked around it with a `FakeSettings` test fixture; manual UAT exposed that the *production* path was equally masked. Resolved in `fdd61a9` by deleting the short-circuit.

**Impact:** Two rounds of band-aid: a test-only `FakeSettings` (Plan 04), then a production refactor (UAT round). The straight line would have been to remove the short-circuit during Plan 04 and let the probe own the signal in both contexts.
**Source:** 04.2-04-SUMMARY.md (Deviations), commit fdd61a9

---

### Worktree branches outlive their agents â€” locks need explicit unlock
After a Claude Code worktree-isolated agent stalls mid-run, `git worktree remove` fails with `cannot remove a locked working tree, lock reason: claude agent agent-XXX (pid YYY)`. Recovery requires `git worktree unlock <path>` first, then `git worktree remove --force`, then `git branch -D`.

**Impact:** Two recovery loops in Wave 1, each costing ~5 minutes of careful state inspection (which file was uncommitted, which branch was ahead, etc.). Salvaged uncommitted partial edits to `/tmp` before unlocking, in case force-remove dropped them.
**Source:** 04.2-02-SUMMARY.md, 04.2-03-SUMMARY.md (Deviations)

---

### lucide-react@1.16.0 is the current major (not 0.x)
`npm install lucide-react` resolved to `1.16.0` against expectations of a `0.5xx`-style version. The package bumped past `1.0` in early 2026; pinning to `^0.x` would have failed.

**Impact:** None â€” caught at install time. Worth noting for future component plans that copy-paste lockfile snippets from older references.
**Source:** Commit 306eb53, frontend/package.json

---

### Both Wave 1 background agents stalled in the same wall-clock window
Plans 02 and 03 launched ~1 second apart and both hit the 600s watchdog within a 90-second window. The cause is correlated load on the SSE layer (large parallel tool_result streams), not random.

**Impact:** A second simultaneous failure during a single recovery means the orchestrator must reason about *two* dirty worktrees + *two* possibly-leaked stashes. The recovery steps doubled.
**Source:** 04.2-02-SUMMARY.md, 04.2-03-SUMMARY.md (both flag the same root cause)
