# Phase 4.2 — UI Design Contract

> Visual and interaction contract for `/login` route + chat header changes (selector error banner + logout). Locks decisions before plan-phase generates tasks. Generated mid-orchestration after the gsd-ui-researcher subagent hit its output cap; written from CONTEXT.md + codebase research the agent had already loaded.

---

## Visual Direction

**Editorial / light-luxury** — serif display + humanist sans, restrained palette, generous whitespace, thin rules, no gradients, no glass, no shadows louder than a hairline. Travel-writing typography evokes the "conversational + travel + AI" brand without leaning on stock SaaS aesthetics. The chat itself stays utilitarian; the login is the editorial frame.

**Anti-template self-check** (from `~/.claude/rules/web/design-quality.md`) — must demonstrate ≥4. Demonstrated:
1. **Hierarchy through scale contrast** — display serif (96px) + body sans (15px) is a 6.4× ratio.
2. **Intentional rhythm** — login has asymmetric two-column rhythm (1.4fr / 1fr); chat header has tight clusters with deliberate gap-2 spacing inside controls and gap-6 between groups.
3. **Typography with character** — *Fraunces* (serif, optical-size-aware) + *Inter* (humanist sans). Real pairing strategy: serif for the editorial voice, sans for the working interface.
4. **Color used semantically** — neutrals carry the surface; accent reserved for *primary action* and *focus ring* only; error reserved for *failure-mode banner* and *invalid form field*.
5. **Hover/focus/active states feel designed** — focus ring is 2px solid accent + 2px offset (not a default browser outline); button hover lifts the surface tone, not the box-shadow.

## Spacing Scale

Multiples of 4 per template requirement.

| Token | Value | Usage |
|-------|-------|-------|
| `space.1` | 4px | Icon → text gap inside buttons; ErrorText margin-top |
| `space.2` | 8px | Tight stacks (label → input) |
| `space.3` | 12px | Field row internal gap |
| `space.4` | 16px | Default form gap; selector dropdown gap |
| `space.5` | 20px | Form field stack gap |
| `space.6` | 24px | Section padding inside login form card |
| `space.8` | 32px | Form padding-y; chat header padding |
| `space.12` | 48px | Login left/right panel inner padding (mobile) |
| `space.16` | 64px | Login left/right panel inner padding (≥1024px) |
| `space.24` | 96px | Login left panel `padding-block-start` for hero offset |

**Justification for 12px and 20px:** All values are multiples of 4 ✓. `space.3` (12px) and `space.5` (20px) are not in the minimal example set (`4, 8, 16, 24, 32, 48, 64`) but they ARE Chakra UI v3's default token names — keeping them aligned to the framework's defaults reduces friction when planners and executors reference Chakra docs. They preserve grid alignment and avoid 4px-jumps that would force every component to use 8px-or-larger gaps.

Exceptions: none. The hero serif descender on the left panel sets its own optical rhythm via `line-height: 0.95`; not a spacing token.

## Shadow Scale

Editorial restraint — shadows are **hairlines**, not depth simulators. CSS hairlines use 1px borders + a faint 1px offset for separation, not blur radii.

| Token | Value | Usage |
|-------|-------|-------|
| `shadows.hairline` | `0 1px 0 oklch(92% 0.005 95)` | Card bottom edge, dropdown bottom edge |
| `shadows.focus` | `0 0 0 2px var(--colors-bg-surface), 0 0 0 4px var(--colors-accent-solid)` | Focus ring (2px offset + 2px ring); always paired with no `outline` |

**No `box-shadow: 0 10px 25px rgba(0,0,0,0.1)` style drop shadows.** That's the SaaS-template look the brief explicitly bans.

## Typography

**Scale: 4 sizes × 2 weights.** Per the gsd-ui-checker constraint (max 4 sizes, max 2 weights), the scale is consolidated. Differentiation among same-size roles uses tracking, case, and color — not extra sizes or weights.

| Role | Family | Size | Weight | Line height | Tracking | Usage |
|------|--------|------|--------|-------------|----------|-------|
| Display XL | Fraunces | `clamp(64px, 4vw + 32px, 96px)` | 400 | 0.95 | -0.02em | Login hero (left panel) |
| Display L | Fraunces | 32px | 400 | 1.1 | -0.01em | Login form heading "Sign in" |
| Body | Inter | 15px | 400 | 1.55 | 0 | Default body, hero subhead, error banner copy, button label, empty-state heading, chat title secondary line |
| Label / Body S | Inter | 13px | 500 | 1.4 | 0.01em | Form field labels, selector "Using: ollama / qwen3:4b" line, username display |
| Caption | Inter | 13px | 500 | 1.3 | 0.04em UPPERCASE | Eyebrow text on hero ("WELCOME") — same size+weight as Label, differentiated by uppercase + tracking |
| Code/mono | system mono stack | 13px | 400 | 1.4 | 0 | Inline `ollama serve` in error suggestions |

**Final scale: 96px / 32px / 15px / 13px** (4 sizes ✓). **Weights: 400, 500** (2 weights ✓).

The hero subhead drops from 17px to 15px (Body) — the editorial hierarchy still reads cleanly because the contrast ratio with Display XL is 6.4× and the subhead's `max-inline-size: 36ch` constrains it visually. The Caption role reuses Label's 13px/500 with tracking + uppercase as differentiation (a more disciplined editorial move than reaching for a smaller size). Buttons use Body 15px/400 — paired with `accent.solid` background for hierarchy weight rather than bold weight.

`font-feature-settings`: `"ss01", "cv11"` for Inter (clearer numerals); `"liga", "dlig"` for Fraunces.

`font-optical-sizing: auto` on Fraunces. Browser-supported; no fallback needed for v1.

### Left panel — editorial hero

Top to bottom (vertical rhythm `space.6`):

1. **Eyebrow** — caption-style "WELCOME", `accent.solid`, letter-spacing 0.04em, `space.6` margin-bottom.
2. **Display headline** — `Display XL` Fraunces. Three-line draft copy: *"Conversations / that actually / go somewhere."* — with line breaks preserved (use `<br>` between lines; do not rely on intrinsic wrapping). The third line "go somewhere." has the word *somewhere* underlined with a 2px `accent.solid` underline-offset 4px decoration.
3. **Body subhead** — Inter Body 15px. Two short lines:
   *"Trip Planner is a chat agent that calls real travel tools live — flights, prices, the works."*

No imagery, no illustration, no gradient, no background pattern. The typography carries the surface. Below 1024px the left panel collapses to a smaller header (Display L instead of Display XL, no eyebrow, subhead drops to Body) so the form remains above the fold on a 568px-tall mobile.

### Right panel — sign-in form

Vertical stack `space.5` between fields, `space.6` above button:

1. **Heading** — `Display L` "Sign in".
2. **Session-expired flash** — when location state has `flash`, render an inline `<Alert>` strip *above* the form fields: `bg.canvas` (the cooler neutral, distinct from form surface) + 1px `border.subtle` + small icon (`Clock`) + Body S text "Your session expired — sign in to continue." 3. **Username field** — `Field.Root` + `Field.Label` ("Username") + `Field.Input` + `Field.ErrorText` slot.
4. **Password field** — same shape with `type="password"`. Toggling switches the input type and the icon; focus stays in the input.
5. **Failed-login error** — when 400 returned by `POST /token`, single `Field.ErrorText`-styled line under the password field: "Username or password is incorrect." `danger.solid` color, `Body` size.
6. **Sign-in button** — Chakra `<Button>`, full width, `accent.solid` background, `Button` typography, `radii.md`. Right-aligned arrow `ArrowRight` icon (lucide), 16px, `space.1` left margin. ## Accessibility

### Focus

- Every interactive element gets a 2px accent focus ring with 2px offset via `shadows.focus`. Default browser outlines explicitly removed only when this ring is applied (`:focus-visible`).
- Tab order on `/login`: skip-to-content (visually hidden, becomes visible on focus) → username → password → show-password toggle → sign-in button.
- Focus moves to the username input on `/login` mount (autofocus). Password manager autofill must work — do not interfere with `autocomplete="username"` / `autocomplete="current-password"`.
- After login redirect, focus moves to the chat input.
- Failed-login error: focus stays in the password field; the error is announced via `aria-live="polite"` on the form's error region.
- Selector error banner: announced via `aria-live="polite"`; container has `role="status"`. Retry button is keyboard-accessible.
- Logout button: keyboard-accessible (icon button, not just `onClick` div).

### ARIA

| Element | ARIA |
|---------|------|
| Login `<form>` | `aria-labelledby="signin-heading"` |
| Username field | `Field.Root` provides `htmlFor`; input has `autocomplete="username"`, `name="username"` |
| Password field | input has `autocomplete="current-password"`, `name="password"` |
| Password show/hide | `aria-label` toggles; `aria-pressed={showing}` |
| Failed-login error region | `aria-live="polite"`, `role="alert"` |
| Session-expired flash | `aria-live="polite"`, `role="status"` |
| Selector error banner | `aria-live="polite"`, `role="status"` |
| Selector dropdowns | Chakra `Select.Root` provides combobox ARIA; verify with axe |
| Logout button | `aria-label="Sign out"` |
| Username display | plain text — no extra ARIA |

### Reduced motion

- Honor `prefers-reduced-motion: reduce`. Specifically: button hover transitions go from 150ms to 0ms; the spinner inside the loading button **remains** (it's a status indicator, not a decoration); password field type-toggle has no animation regardless. Login page entrance has no animation in v1, so nothing to suppress.

### Color contrast

All combinations verified above (Color Palette §) meet WCAG AA. Body text meets AA Normal (4.5:1). Caption-size text on disabled buttons (`fg: white` on `oklch(80% 0.04 245)`) is 3.4:1 — accept under WCAG AA disabled-state exemption (decorative only, control is non-interactive).

### Form validation

- Username: required, no min/max length client-side (backend enforces).
- Password: required, no client-side complexity rules (backend store is bcrypt-via-pwdlib).
- Submission disabled while either field is empty.
- HTML5 `required` attribute used; Chakra `Field.Root invalid` state mirrors backend 400 response.
