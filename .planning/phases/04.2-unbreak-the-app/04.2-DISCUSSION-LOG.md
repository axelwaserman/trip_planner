## Token Storage + 401 Expiry

User initially declined the three single-question form and asked for a comparison table covering edge cases (multiple tabs, cookies disabled, XSS, CSRF, Vite HMR, Phase 5 throwaway risk). Comparison table provided; user accepted Claude's recommendation as a single bundle.

### Where the JWT lives

| Option | Description | Selected |
|--------|-------------|----------|
| `localStorage` under `auth_token` | Mirrors existing `llm_provider_config`. Survives reloads. XSS-exposed; acceptable for v1 since Phase 8 adds CSP. | ✓ |
| `sessionStorage` under `auth_token` | Per-tab lifetime; user re-logs every new tab. Same XSS exposure. | |
| In-memory only (React state) | Most XSS-resistant. F5 = re-login. Bad for Vite HMR / page reloads. | |
| httpOnly secure cookie | Web-security gold standard. Requires backend `Set-Cookie` + CSRF middleware + `credentials: 'include'` everywhere — expands Phase 4.2 scope significantly. | |

### 401 expiry handling

| Option | Description | Selected |
|--------|-------------|----------|
| Redirect to `/login` + flash message | Centralized fetch wrapper detects 401, clears token, redirects with "Session expired" flash. Loses in-progress chat input. | ✓ |
| Refresh-token flow | New `/refresh` endpoint, refresh-token storage, 401 interceptor → retry. ~3x the auth code surface; Phase 5 will likely redesign anyway. | |
| Inline error, user-driven | 401 surfaces as inline alert with [Re-login] button. More forgiving but more code paths. | |

### Storage layout

| Option | Description | Selected |
|--------|-------------|----------|
| Separate keys: `auth_token` + `llm_provider_config` | Logout clears auth, prefs persist. Matches existing key. | ✓ |
| Namespaced: `trip_planner.auth` + `trip_planner.prefs` | Future-proof; requires renaming existing key. | |
| Single combined object: `trip_planner_state` | Atomic reads/writes; logout has to merge-update — bug-prone. | |

**User's choice:** localStorage + redirect-to-/login on 401 + separate keys (Claude's recommendation, accepted with "I agree with your recommendation, proceed").
**Notes:** XSS exposure of localStorage acknowledged; Phase 8's CSP + sanitization mitigates. Refresh-token flow rejected as scope creep contradicting "quick-and-dirty." Stale-tab problem (logout in tab A leaves tab B with old token) accepted as known limitation — next 401 cleans up.

## Selector Failure-Mode UX

### Error surface

| Option | Description | Selected |
|--------|-------------|----------|
| Inline banner above chat input | Persistent red/amber banner above input. Most discoverable. | |
| Toast / Chakra Toaster | Pops up briefly then dismisses. Bad for persistent misconfig. | |
| Inline message in chat thread | Special `error` role in chat. Scrolls away with conversation. | |
| Dropdown-adjacent inline (under selector) | Small error text directly under provider/model dropdown in header. Tightly coupled to control. | ✓ |

### Error timing

| Option | Description | Selected |
|--------|-------------|----------|
| Proactive: ping provider on session create | Backend probes at selection time. ~200ms latency. User knows before typing. | ✓ |
| Reactive: only on chat send | Wait for first chat to fail. User types long message before learning provider is broken. | |
| Hybrid: API key validate upfront, Ollama lazy | Compromise. | |

### Recovery

| Option | Description | Selected |
|--------|-------------|----------|
| Suggestion text only | "Run `ollama serve` and refresh, or pick another provider." No button. | |
| Suggestion + Retry button | Banner has [Retry] that re-invokes the failing call. | ✓ |
| Suggestion + auto-fallback | Switch to a working provider automatically. Hides failure; could trigger unintended cloud billing. | |

**User's choice:** Dropdown-adjacent inline + proactive probe + Retry button.
**Notes:** Single error surface for all 4 failure buckets (Ollama unreachable, model not pulled, cloud API key missing, /api/providers fetch failed). Selector remains live — user can pick another provider without clicking Retry. Auto-fallback explicitly rejected to avoid silent cloud billing.
