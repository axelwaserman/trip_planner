<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend SelectorErrorBanner → rendered copy | Error copy rendered as text + code elements, never innerHTML |
| useChat apiFetch → /api/chat/session | Session-create request carries Authorization: Bearer; error body parsed as known shape |
| useChat apiFetch → /api/chat SSE | Chat request carries Authorization: Bearer; 401 triggers clearToken + redirect |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-06-01 | Information Disclosure | SelectorErrorBanner renders probe error copy including shell commands | accept | Error copy is server-provided structured text rendered as Chakra Text/code elements (not dangerouslySetInnerHTML). inlineCode items are rendered as <code> text. No HTML injection vector. RESEARCH.md §"Known Threat Patterns" confirms this is safe. |
| T-42-06-02 | Information Disclosure | useChat apiFetch sends auth token on /api/chat SSE calls | accept | Required behavior (D-04). The token is sent in the Authorization header, not in the URL or body. HTTPS in production. XSS risk is the localStorage exposure documented in D-01 and accepted for v1. |
| T-42-06-03 | Tampering | Retry button re-fires session-create with the same provider/model | accept | User chose the provider; retry is user-initiated. No automatic fallback to a different provider (D-08 explicitly prohibits this). |
| T-42-06-04 | Denial of Service | F4 banner (providers fetch failed) could cause repeated apiFetch retries | mitigate | Retry button is manual (user clicks). apiFetch does not auto-retry. The Retry handler in ChatInterface calls handleProviderChange(currentProvider, currentModel) exactly once per click. |
</threat_model>
