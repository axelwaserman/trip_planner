# Phase 4.2: Unbreak the App - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

A fresh user opens the app in a browser, lands on a `/login` page, signs in with `AUTH_USERS`-seeded credentials, picks an LLM provider+model from a dropdown, and completes one chat turn end-to-end — without hitting silent 401s, without seeing generic "Sorry, I encountered an error", and without the model selector producing a broken session.

Two requirements close the loop:
- `REQ-login-page` — the missing frontend half of `REQ-auth-backend`. JWT-auth surface (login route, protected-route wrapper, auth-aware fetch, logout) so `/api/*` calls actually carry an `Authorization: Bearer` header.
- `REQ-llm-provider-ui-fix` — the broken wiring of `REQ-llm-provider-ui-config`. Provider/model failures (Ollama unreachable, model not pulled, cloud API key missing, `/api/providers` failure) surface as deterministic, recoverable errors at selection time.

**In scope:** `/login` route, protected-route wrapper, auth-aware fetch, logout, JWT in localStorage, 401 redirect, provider health probe, dropdown-adjacent inline error UX with retry.
**Out of scope (deferred):** Postgres-seeded users (Phase 5), refresh tokens, password reset, registration, MFA, httpOnly cookie migration, CSP/sanitization (Phase 8), real cloud LLM providers (Phase 4.5).

</domain>

<decisions>
## Implementation Decisions

### Auth Storage & Lifecycle

- **D-01: JWT lives in `localStorage` under key `auth_token`.**
  Mirrors the existing `llm_provider_config` localStorage pattern. Survives reloads (good for Vite HMR during dev). XSS exposure is acknowledged and accepted for v1 — Phase 8 adds CSP + sanitization, Phase 5 may re-platform auth entirely. httpOnly-cookie migration would be a meaningful re-architecture (backend `Set-Cookie`, CSRF middleware, `credentials: 'include'`, Vite proxy cookie passthrough) and contradicts the "quick-and-dirty" brief.

- **D-02: 401 from any `/api/*` call → clear token, redirect to `/login`, flash "Session expired — please sign in again."**
  Centralized in the fetch wrapper. Loses in-progress chat input on redirect; that's accepted for v1. No refresh-token flow in v1 — that's an auth-protocol expansion Phase 5 will likely redesign anyway.

- **D-03: Storage layout — separate keys: `auth_token` + `llm_provider_config`.**
  Logout clears `auth_token`; `llm_provider_config` persists across logins so the user's last provider/model selection is remembered. Two independent reads on bootstrap.

- **D-04: Token sent as `Authorization: Bearer <jwt>` header on every `/api/*` request.**
  Matches the existing `OAuth2PasswordBearer` backend (`backend/app/api/routes/auth.py`). The fetch wrapper is the single point that injects this.

- **D-05: Logout clears `auth_token` and any in-flight chat session, then routes to `/login`.**
  Does NOT clear `llm_provider_config` (per D-03).

### Selector Failure-Mode UX

- **D-06: Errors render under the provider/model selector in the chat header (dropdown-adjacent inline).**
  Single error surface for all four failure modes (Ollama unreachable, model not pulled, cloud API key missing, `/api/providers` failure). Tightly coupled to the control causing the error. Persistent — does not auto-dismiss. Header stays visible while the chat scrolls, so the message remains visible while the user reads context.

- **D-07: Errors fire proactively at session-create time (not lazily on first chat send).**
  Backend probes provider liveness as part of `POST /api/chat/session`:
  - Ollama: hit `GET {ollama_base_url}/api/tags`, verify selected model is in the list.
  - Cloud (OpenAI/Anthropic): currently keyed on env-var presence only — no live key validation in v1; that's Phase 4.5's `validate_config` deliverable.
  - User knows the provider is broken before typing a message.
  - Cost: ~200ms latency on selection. Acceptable.

- **D-08: Recovery UX is "suggestion text + Retry button + selector remains live."**
  Banner reads e.g. `⚠ Ollama is unreachable at http://localhost:11434. [ Retry ]` — Retry re-invokes the failing call (re-fetches `/api/providers` or re-creates the session). The dropdown stays interactive so the user can also pick a different provider/model without clicking Retry. No auto-fallback to a different provider (would hide the failure and could trigger unintended cloud billing).

- **D-09: Failure-mode message taxonomy (final copy lives in UI-SPEC.md, but the buckets are locked):**
  - **F1 — Ollama unreachable:** `Can't reach Ollama at {OLLAMA_BASE_URL}. Run \`ollama serve\` and retry, or pick another provider.`
  - **F2 — Selected model not pulled:** `Ollama is running but {model} isn't installed. Run \`ollama pull {model}\` or pick a different model.`
  - **F3 — Cloud API key missing:** `{Provider} needs an API key. Set {ENV_VAR} on the backend and restart.`
  - **F4 — `/api/providers` fetch failed:** `Couldn't load providers. Check that the backend is running. [ Retry ]`

### Login Visual Direction

- **D-10: Editorial split-screen for `/login`.**
  Hero-style left side ("Trip Planner. Conversational travel tools." or similar — UI-SPEC.md owns final copy/typography/atmosphere) + sign-in form on the right. Treats login as the product's first frame, not a default Chakra centered card. Anti-template policy from `~/.claude/rules/web/design-quality.md` is in force — UI-SPEC.md will spec the typography pairing, palette, and any subtle motion. Phase 4.2 ships a believable v1 of this; later phases can polish.

- **D-11: Form uses Chakra v3 `Field.Root` + `Field.Label` + `Field.ErrorText` per the project's chakra-ui skill, not raw `<input>` with hand-rolled state.**
  Compound-component pattern — matches the existing codebase (`Collapsible.Root`, etc.).

- **D-12: Password field uses an unmasked-toggle (eye icon) for usability.**
  Standard pattern, no debate.

- **D-13: Failed-login error renders inline under the form ("Incorrect username or password"), backend's existing 400 message text.**
  No toast for login errors — they happen on the same screen the user is looking at.

### Routing & Protected Routes (Claude's Discretion)

- **D-14: Use `react-router-dom` v7 (`BrowserRouter` + `Routes` + `Route`).**
  Boring, ubiquitous, has a working v7 release. No router today; this is the obvious add. If you'd rather use `@tanstack/react-router` or roll your own, flag at planning time.

- **D-15: `<RequireAuth>` wrapper component checks `auth_token` and `getCurrentUser()`; redirects to `/login` with a `?from=` param so post-login can return to the original destination.**
  Standard pattern. Composes once around `<ChatInterface />`.

- **D-16: Routes: `/login` → `<Login />`, `/` → `<RequireAuth><ChatInterface /></RequireAuth>`. Default redirect from `/` to `/login` when unauthed (handled by `<RequireAuth>`).**
  No deeper hierarchy needed in v1.

### Backend additions (minimal)

- **D-17: New `GET /api/auth/me` endpoint returning the current user (uses existing `get_current_active_user`).**
  Frontend uses this on page load to verify the stored token is still valid; if it 401s, clear and redirect. Avoids decoding the JWT client-side.

- **D-18: Provider health probe exposed via `POST /api/chat/session` itself, not a new endpoint.**
  When session-create receives `{provider, model}`, run the liveness check before allocating the session. On failure: 502 (`Ollama unreachable`) or 400 (`Model not installed`) with a structured error body `{ error: "ollama_unreachable" | "model_not_installed" | "missing_api_key", message, hint }` so the frontend can map error codes to localized strings.

- **D-19: `AUTH_USERS` env-seed stays as the user source.**
  No changes to `backend/app/api/routes/auth.py` user store. Phase 5 retires it.

### Claude's Discretion

- File layout under `frontend/src/`: `pages/Login.tsx`, `lib/auth.ts` (token getters, fetch wrapper), `hooks/useAuth.ts` or `contexts/AuthContext.tsx` — researcher should pick what fits the existing codebase shape.
- The exact split-screen visual treatment (typography, palette, copy) belongs to UI-SPEC.md.
- Test split: backend gets unit tests for the new probe + endpoint; frontend tests follow existing Vitest patterns. E2E for the login flow is Phase 4.3's job (see `REQ-ci-reset` deliverables).
- Whether to write a `<TokenExpiredAlert />` component vs. inlining the redirect logic — implementation detail.
- Any ARIA wiring beyond what Chakra `Field.Root` provides by default.

</decisions>
