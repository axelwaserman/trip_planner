# Plan 04.2-04 Summary — be-session-probe-wire

## Outcome

`POST /api/chat/session` now runs `probe_provider()` after the existing provider/model
validation and before `chat_service.create_session()`. On `ProbeError`, the route raises
`HTTPException` with `detail=probe.model_dump()` (a structured `{error, message, hint}`
body the frontend's `mapProbeError` consumes) and the status code mapping from RESEARCH.md
Pitfall 4: `502` for `ollama_unreachable`, `400` for `model_not_installed` and `missing_api_key`.

## Deviations

**Test scaffolding for missing_api_key.** The existing `available=False` 400 path in
`create_session` fires when `openai_api_key` is `None`, returning the older string-detail
body. To exercise the probe's structured `missing_api_key` detail in the integration test,
the test substitutes a `FakeSettings` object on both `routes.settings` and
`provider_probe.settings` that reports openai as available — making the probe the authority
for the missing-key signal. No production code changed; this is test-only and aligns with
the plan's intent that the probe own the F1-F4 taxonomy.
